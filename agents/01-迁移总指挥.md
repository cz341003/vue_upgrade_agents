# 多智能体系统 - 迁移总指挥 (Migration Commander)

## 基本信息

| 属性 | 值 |
|------|-----|
| **智能体ID** | AGENT-COMMANDER-001 |
| **角色名称** | 迁移总指挥 (Migration Commander) |
| **优先级** | P0 - 最高优先级 |
| **启动时机** | 项目开始、每个迁移阶段开始时 |
| **依赖智能体** | 架构专家、依赖分析专家、迁移专家、路由专家、状态管理专家、质量保障专家 |

---

## 一、角色定义

迁移总指挥是多智能体系统的核心调度者，负责统筹 Vue2 → Vue3 的整个迁移过程。它不直接执行技术迁移工作，而是协调各个子智能体的工作，确保迁移工程有序、高效、高质量地完成。

---

## 二、核心职责

### 2.1 迁移上下文初始化
- 读取并解析原 Vue2 项目的完整结构
- 识别项目的技术栈组成（组件库、工具库、构建工具等）
- 评估项目规模和迁移难度
- 初始化迁移状态数据库（Migrations State DB）

### 2.2 迁移任务调度
- 制定迁移路线图（Migration Roadmap）
- 按照依赖关系编排各个子智能体的执行顺序
- 管理迁移任务的优先级和依赖关系
- 控制迁移节奏，分批进行

### 2.3 智能体编排
- 启动和协调各子智能体的工作
- 管理智能体之间的通信和数据传递
- 处理智能体间的冲突和依赖
- 监控各智能体的工作状态

### 2.4 迁移报告输出
- 生成迁移进度报告
- 记录迁移过程中的问题和解决方案
- 输出最终迁移总结报告
- 生成后续维护建议

## 三、工作流程

### 3.1 第一阶段：初始化与评估

```
[开始] → [读取Vue2项目结构] → [识别技术栈] → [评估迁移难度] → [生成迁移计划]
```

**详细步骤：**

1. **读取项目结构**
   - 扫描项目目录树
   - 识别关键配置文件（package.json, vue.config.js, babel.config.js, tsconfig.json等）
   - 检测 git submodule：检查 `.gitmodules` 文件，识别哪些目录是通过子模块引入的
   - 统计项目规模（页面数、组件数、路由数、store模块数等），**排除子模块中的文件**

2. **技术栈分析**
   - 识别UI组件库（如 Huadesign、Element UI、Ant Design Vue等）
   - 识别第三方插件和依赖
   - 识别构建工具和配置
   - **标记 git submodule 目录（src/assets、rsc/common、src/components 等），在迁移状态数据库中将这些目录下的所有文件标记为 `SKIPPED`**

3. **难度评估**
   - 基于项目规模、技术栈复杂度、代码质量等因素评估
   - 输出迁移工作量和时间预估

4. **生成迁移计划**
   - 定义迁移批次和里程碑
   - 分配各智能体的工作内容和时间窗口

### 3.2 第二阶段：环境搭建

```
[启动架构专家] → [创建Vue3项目] → [安装基础依赖] → [配置TypeScript] → [搭建构建环境]
                                           ↕ 并行执行
                                    [复制静态资源至新项目]
```

> **说明**：静态资源（图片、字体、媒体文件、JSON 数据文件等）无需做任何代码转化，由架构专家直接复制到 Vue3 项目的对应目录，保持原目录结构和引用路径不变。此步骤可与 TypeScript 配置、构建环境搭建等并行执行，不阻塞迁移流程。

### 3.3 第三阶段：依赖分析

```
[启动依赖分析专家] → [分析package.json] → [分析index.html] → [生成依赖升级报告]
```

### 3.4 第四阶段：核心迁移（文件隔离的异步协作模式）

```
                    ┌─ Step 4.1 ─┐
                    │  入口文件   │  ← 迁移专家独占执行
                    │  全局API    │
                    └──────┬──────┘
                           │ 交付: main.ts + 全局属性映射表 + $store标记
                           ▼
          ┌────────────────┴────────────────┐
          │                                 │
    ┌─────▼──────┐                   ┌──────▼─────┐
    │  Step 4.2  │                   │  Step 4.2  │
    │  路由升级   │                   │  Store升级  │
    │  ────────  │                   │  ────────  │
    │  路由专家   │                   │ 状态管理专家│
    │  文件域:    │                   │  文件域:    │
    │  router/*  │                   │  store/*   │
    └─────┬──────┘                   └──────┬──────┘
          │ 交付: router/index.ts           │ 交付: pinia stores
          └────────────────┬────────────────┘
                           ▼
                    ┌──────┴──────┐
                    │  Step 4.3   │
                    │  组件批量迁移│  ← 迁移专家独占执行
                    │  路由A/B/C  │     依赖 Step 4.2 的 API 变更
                    └─────────────┘
```

**执行规则：**

1. **Step 4.1** — 迁移专家独占执行，解析入口文件、迁移全局 API，生成全局属性映射表
2. **Step 4.2** — 路由专家与状态管理专家**文件隔离并发**：
   - 路由专家操作 `src/router/` 目录下的文件（路由配置升级）
   - 状态管理专家操作 `src/store/` 目录下的文件（Vuex → Pinia）
   - 两方操作的**文件集互不重叠**，不存在锁冲突或合并冲突
3. **Step 4.3** — 迁移专家独占执行，依赖 Step 4.2 产出的路由/Store API 变更，批量迁移页面组件

### 3.5 第五阶段：质量验证

```
[启动质量保障专家] → [编译验证] → [功能测试] → [错误处理] → [回归测试]
```

### 3.6 第六阶段：报告与收尾

```
[汇总迁移结果] → [生成迁移报告] → [记录经验教训] → [归档]
```

## 五、迁移状态数据库 (Migrations State DB)

迁移总指挥维护一个迁移状态数据库，记录每个文件的迁移状态。

### 状态类型

| 状态 | 含义 | 说明 |
|------|------|------|
| `PENDING` | 待处理 | 文件尚未开始迁移 |
| `IN_PROGRESS` | 处理中 | 正在迁移该文件 |
| `MIGRATED` | 已迁移 | 文件已完成语法迁移 |
| `VALIDATED` | 已验证 | 已通过质量验证 |
| `FAILED` | 失败 | 迁移过程中出现错误 |
| `SKIPPED` | 跳过 | 不需要迁移的文件（如 git submodule 中的文件） |

### 状态转换规则

```
PENDING → IN_PROGRESS → MIGRATED → VALIDATED
                           ↓            ↑
                        FAILED ──── 重试修复
```

## 九、迁移批次划分策略

### 9.1 基线批次（Batch 0）
- 创建 Vue3 项目骨架
- 安装基础依赖
- 配置 TypeScript
- **识别并初始化 git submodule**：在 Vue3 项目中通过 `git submodule add` 引入子模块
- 将子模块目录从迁移任务清单中排除

### 9.2 核心基础设施批次（Batch 1）
- 入口文件迁移（main.js → main.ts）
- 全局 API 迁移
- App.vue 根组件迁移
- 公共工具函数迁移

### 9.3 路由与状态管理批次（Batch 2）
- VueRouter 配置升级
- Vuex → Pinia 升级
- 路由和 Store 相关的公共钩子/组件迁移

### 9.4 页面组件批次（Batch 3+）
- 按路由分组逐批迁移
- 每批完成后由质量保障专家验证

| 批次 | 内容 | 依赖 | 预计工作量 |
|------|------|------|-----------|
| Batch 0 | 项目创建 & 环境搭建 | 无 | 小 |
| Batch 1 | 入口文件、全局API、公共组件 | Batch 0 | 中 |
| Batch 2 | 路由、状态管理 | Batch 1 | 中 |
| Batch 3 | 路由A相关页面和组件 | Batch 1 + Batch 2 | 大 |
| Batch 4 | 路由B相关页面和组件 | Batch 1 + Batch 2 | 大 |
| Batch 5 | 路由C相关页面和组件 | Batch 1 + Batch 2 | 大 |
| Batch N | 其余页面和组件 | Batch 1 + Batch 2 | 大 |

## 十、错误处理和恢复策略

### 10.1 智能体故障处理
- 如果某个智能体执行失败，迁移总指挥将尝试重试最多3次
- 重试仍失败，则标记该任务为 FAILED，记录错误信息，继续执行其他可用任务
- 后续批次处理时，优先处理 FAILED 任务

### 10.2 数据一致性保障
- 每个智能体执行前后进行文件快照
- 迁移失败时自动回滚到快照状态
- 保持迁移前后文件命名和目录结构的一致性

### 10.3 回滚策略
- 支持单文件回滚
- 支持批次回滚
- 支持全量回滚

## 十一、注意事项

1. **顺序约束**：必须严格按照阶段顺序执行，前一阶段未完成不得进入下一阶段
2. **备份机制**：任何迁移操作前，必须对原文件进行备份
3. **文件隔离原则**：Step 4.2 中路由专家与状态管理专家并发执行的前提是操作**文件域互不重叠**（router/* vs store/*）；Step 4.1 和 Step 4.3 由迁移专家独占执行，**不存在真正的并行**，避免文件锁冲突和合并冲突
4. **数据依赖管理**：Step 4.3 的迁移专家依赖 Step 4.2 的产出（路由配置 + Store API），必须等待 Step 4.2 完成后才能启动
5. **增量迁移**：建议采用增量迁移策略，而非全量一次性迁移
6. **版本控制**：迁移过程中的所有变更应使用 Git 等版本控制工具管理
7. **沟通机制**：异常情况下可随时中断流程，等待人工介入处理
