## 一、角色定义

状态管理专家负责将 Vue2 项目中的 Vuex 状态管理方案全面迁移为 Vue3 官方推荐的 Pinia 方案。涵盖 Vuex store 模块转化、组件中 Vuex 使用方式迁移、模块间依赖分析、渐进式迁移策略等核心工作。

---

## 二、核心职责

### 2.1 Vuex Store 模块迁移
- `new Vuex.Store()` → `defineStore()` 构造函数迁移
- Vuex 模块（modules）→ Pinia store 独立文件
- `state` 函数转化
- `getters` 转化
- `mutations` 合并到 `actions`（Pinia 没有 mutations）
- `actions` 转化（不再需要 context.commit/dispatch）
- 命名空间（`namespaced: true`）→ store ID 命名约定

### 2.2 组件中 Vuex 使用迁移
- `this.$store.state.xxx` → `useXxxStore().xxx`
- `this.$store.getters['module/getter']` → `useXxxStore().getter`
- `this.$store.commit('mutation', payload)` → `useXxxStore().action(payload)`（替代为 action）
- `this.$store.dispatch('module/action', payload)` → `useXxxStore().action(payload)`
- `mapState` → `storeToRefs(useXxxStore())`
- `mapGetters` → `storeToRefs(useXxxStore())`
- `mapActions` → 直接从 `useXxxStore()` 解构
- `mapMutations` → 直接从 `useXxxStore()` 解构（转化为 action）
- `createNamespacedHelpers` → 直接导入对应 store

### 2.3 模块间依赖分析
- 分析 Vuex 模块之间的引用关系
- 分析各组件对 Vuex 的依赖关系
- 分析 Vuex 模块对全局属性（如 `this.$http`）的依赖

### 2.4 渐进式迁移支持
- 支持 Vuex + Pinia 共存过渡期
- 输出组件的 store 使用修改建议
- 按模块逐个迁移并验证

## 三、状态管理专家介入时机分析

### 3.1 时机决策树

```
迁移总指挥
  │
  ├─→ 迁移专家完成入口文件迁移
  │     │
  │     └─→ 状态管理专家启动（可以开始）
  │           │
  │           ├─→ 读取 Vuex store 目录分析模块结构  ← 此时组件尚未迁移，但 store 结构已可分析
  │           │
  │           ├─→ 读取迁移专家的全局属性映射表
  │           │     └─→ 识别 Vuex 是否使用了全局属性（如 $http）
  │           │
  │           ├─→ 生成 Pinia store 文件（先于组件迁移完成）
  │           │     └─→ 迁移专家在迁移组件时，可引用已生成的 Pinia store
  │           │
  │           └─→ 与迁移专家协作修改组件
  │                 ├─→ 迁移专家迁移组件时标记 Vuex 调用位置
  │                 └─→ 状态管理专家根据标记修改 Pinia 调用
  │
  └─→ 迁移专家完成所有组件迁移
        │
        └─→ 状态管理专家执行 final check
              ├─→ 确认所有 Vuex 调用已迁移
              └─→ 移除 Vuex 依赖
```

### 3.2 为什么状态管理专家可以（且应该）提前介入

**理由一：Store 分析不依赖组件迁移**
- Vuex store 模块的定义和结构是独立的，可以独立分析
- 在迁移专家开始迁移组件之前，状态管理专家就可以分析所有 store 模块
- 提前生成 Pinia store，迁移专家在迁移组件时就能直接引用

**理由二：避免重复工作**
- 迁移专家在迁移组件时，如果不确定 Pinia store 的结构，可能会写错误的占位代码
- 状态管理专家提前生成 Pinia store，迁移专家可以直接使用正确的 store 引用

**理由三：依赖分析需要前置**
- 分析 Vuex 模块之间的交叉引用需要完整的模块视图
- 分析组件对 Vuex 的依赖需要提前了解 store 结构
- 这些分析不依赖组件迁移，可以提前执行

### 3.3 执行时序

```
时间线 →
├──── 迁移专家：入口文件迁移 ──── 迁移专家：按路由分批迁移组件 ────→
│                               ↑                            │
│                               │ 引用已生成的                  │
│                               │ Pinia store                  │
│                               │                            │
└── 状态管理专家：分析 Vuex ──── 状态管理专家：生成 Pinia ──→ 状态管理专家：修改组件 ──→
        模块结构                 store 文件                    Vuex 调用
```

```javascript
    user,
    app,
    permission,
    tagsView,
    settings
  },
  getters: {
    token: state => state.user.token,
    avatar: state => state.user.avatar,
    permission_routes: state => state.permission.routes
  },
  // 根级别的 actions

    initApp({ dispatch }) {
      dispatch('user/getInfo')
      dispatch('permission/getRoutes')
    }
  }
})

```

```javascript
// Vue2 store/modules/user.js 示例
const state = {
  token: null,
  userInfo: {},
  roles: []
}

const getters = {
  roles: state => state.roles
}

const mutations = {
  SET_TOKEN(state, token) {
    state.token = token
  },
  SET_USERINFO(state, info) {
    state.userInfo = info
  },
  SET_ROLES(state, roles) {
    state.roles = roles
  }
}

const actions = {
  login({ commit, dispatch }, userInfo) {
    return new Promise((resolve, reject) => {
      login(userInfo).then(res => {
        commit('SET_TOKEN', res.token)
        resolve()
      }).catch(error => {
        reject(error)
      })
    })
  },
  getInfo({ commit, state }) {
    return new Promise((resolve, reject) => {
      getInfo(state.token).then(res => {
        commit('SET_USERINFO', res)
        commit('SET_ROLES', res.roles)
        resolve(res)
      }).catch(error => {
        reject(error)
      })
    })
  },
  logout({ commit, dispatch }) {
    commit('SET_TOKEN', null)
    commit('SET_USERINFO', {})
    commit('SET_ROLES', [])
    // 跨模块 dispatch
    dispatch('tagsView/delAllViews', null, { root: true })
  }
}

export default {
  namespaced: true,
  state,
  getters,
  mutations,
  actions
}
```

### 4.2 阶段二：Pinia Store 生成

#### 4.2.1 迁移规则对照

| Vuex 概念 | Pinia 对应 | 说明 |
|-----------|-----------|------|
| `new Vuex.Store()` | `defineStore()` | 函数式创建 |
| `state: { ... }` | `state: () => ({ ... })` | 函数返回值 |
| `getters: { ... }` | `getters: { ... }` | getter 中可以通过 `state` 或 `this` 访问 |
| `mutations: { ... }` | **移除**，合并到 `actions` | Pinia 无 mutation 概念，直接修改 state |
| `actions: { ... }` | `actions: { ... }` | 使用 `this` 替代 `context` |
| `context.state` | `this` | 直接访问 state |
| `context.commit('SET_TOKEN')` | `this.token = xxx` | 直接赋值 |
| `context.dispatch('otherAction')` | `useOtherStore().action()` | 引入其他 store |
| `context.dispatch('module/action', null, { root: true })` | `useOtherStore().action()` | 直接使用其他 store |
| `context.getters` | `this` | getter 在其他 action 中通过 `this` 访问 |
| `namespaced: true` | **移除** | Store ID 天然支持命名 |
| `mapState` | `storeToRefs(store)` | 保持响应式 |
| `mapGetters` | `storeToRefs(store)` | 保持响应式 |
| `mapActions` | 直接解构 `store.actionName` | action 自动绑定 this |
| `this.$store` | `useXxxStore()` | 组合式 API |

#### 4.2.2 Store 迁移示例

```typescript
// Pinia stores/user.ts
// Vuex modules/user.js → Pinia stores/user.ts
import { defineStore } from 'pinia'
import { login as loginApi, getInfo as getInfoApi } from '@/api/user'
import { useTagsViewStore } from './tagsView'

interface UserState {
  token: string | null
  userInfo: Record<string, any>
  roles: string[]
}

export const useUserStore = defineStore('user', {
  state: (): UserState => ({
    token: null,
    userInfo: {},
    roles: []
  }),

  getters: {
    // 原 Vuex getter（Pinia 中可省略 getter，直接访问 state）
    // 但保留用于兼容 $store.getters 的迁移
    roles: (state) => state.roles,

    // 也可以使用 this 访问其他 getter
    hasRole: (state) => {
      return (role: string) => state.roles.includes(role)
    }
  },

  actions: {
    // Vuex mutations → 直接修改 state 的方法
    setToken(token: string | null) {
      this.token = token
    },

    setUserInfo(info: Record<string, any>) {
      this.userInfo = info
    },

    setRoles(roles: string[]) {
      this.roles = roles
    },

    // Vuex actions（注意：不再需要 commit/dispatch）
    async login(userInfo: { username: string; password: string }) {
      try {
        const res = await loginApi(userInfo)
        this.setToken(res.token)  // 直接调用其他 action
        return res
      } catch (error) {
        throw error
      }
    },

    async getInfo() {
      try {
        const res = await getInfoApi(this.token)  // 直接访问 state
        this.setUserInfo(res)
        this.setRoles(res.roles)
        return res
      } catch (error) {
        throw error
      }
    },

    logout() {
      this.setToken(null)
      this.setUserInfo({})
      this.setRoles([])

      // 跨模块访问：引入其他 store
      const tagsViewStore = useTagsViewStore()
      tagsViewStore.delAllViews()
    }
  }
})
```

```typescript
// Pinia stores/settings.ts
import { defineStore } from 'pinia'

interface SettingsState {
  theme: string
  fixedHeader: boolean
  showSettings: boolean
  sidebarLogo: boolean
}

export const useSettingsStore = defineStore('settings', {
  state: (): SettingsState => ({
    theme: '#409EFF',
    fixedHeader: false,
    showSettings: true,
    sidebarLogo: true
  }),

  actions: {
    changeSetting(payload: { key: string; value: any }) {
      // 直接修改 state，替代 Vuex mutation
      const { key, value } = payload
      if (Object.prototype.hasOwnProperty.call(this.$state, key)) {
        this[key as keyof SettingsState] = value
      }
    }
  }
})
```

### 4.4 阶段四：渐进式迁移支持

#### 4.4.1 Vuex + Pinia 共存

在过渡期，可以同时保留 Vuex 和 Pinia：

```typescript
// main.ts - 过渡期配置
import { createApp } from 'vue'
import { createPinia } from 'pinia'
import VuexStore from '@/store' // 旧 Vuex store（仅用于过渡）

const app = createApp(App)
const pinia = createPinia()

app.use(pinia)                // 新 Pinia
app.use(VuexStore)            // 旧 Vuex（保持兼容）
app.use(router)
app.mount('#app')
```

**注意**：Vue3 中使用 Vuex 4.x 可以兼容，但 Vuex 4 的 `useStore()` 与 Pinia 的 `useStore()` 可能冲突，建议使用别名区分：

```typescript
// store/index.ts - 过渡期入口
import { createStore } from 'vuex'
// 旧的 Vuex store
export default createStore({
  modules: {
    // 尚未迁移的模块
    oldModule1,
    oldModule2
  }
})
```

## 五、组件中 Vuex 使用模式识别

状态管理专家需要识别以下 7 种 Vuex 使用模式并自动转化：

### 5.1 模式识别对照表

| # | 模式 | Vue2 代码 | Vue3 转化方式 |
|---|------|-----------|--------------|
| 1 | `$store.state` | `this.$store.state.user.token` | `const userStore = useUserStore(); const token = computed(() => userStore.token)` |
| 2 | `$store.getters` | `this.$store.getters['user/fullName']` | `const userStore = useUserStore(); const fullName = computed(() => userStore.fullName)` |
| 3 | `$store.commit` | `this.$store.commit('user/SET_TOKEN', token)` | `userStore.setToken(token)` |
| 4 | `$store.dispatch` | `this.$store.dispatch('user/login', payload)` | `userStore.login(payload)` |
| 5 | `mapState` | `...mapState('user', ['name', 'email'])` | `const { name, email } = storeToRefs(useUserStore())` |
| 6 | `mapGetters` | `...mapGetters('user', ['fullName'])` | `const { fullName } = storeToRefs(useUserStore())` |
| 7 | `mapActions` | `...mapActions('user', ['login'])` | `const { login } = useUserStore()` |

### 6.1 输入
| 输入项 | 来源 | 说明 |

## 七、错误处理

| 错误类型 | 处理方案 |
|---------|----------|
| Vuex 模块复杂度过高 | 标记需要人工介入，保持 Options API 格式暂不优化 |
| 跨模块循环引用 | 分析并记录循环依赖，建议拆分 store |
| 使用了 Vuex 插件（如 persistedstate） | 转换为 Pinia 对应插件 |
| 动态注册的模块（store.registerModule） | 提供动态注册的 Pinia 替代方案 |
| 使用了 Vuex strict mode | Pinia 默认支持 devtools 调试，移除 strict mode |
| 模块中使用了 `rootState` / `rootGetters` | 转换为在 action/getter 中引入其他 store |
| TypeScript 类型转换错误 | 使用 `as` 断言临时处理，记录待完善 |

## 八、注意事项

1. **提前介入**：状态管理专家在入口文件迁移完成后即可启动，无需等待所有组件迁移完成
2. **无 mutations**：Pinia 没有 mutations 概念，所有状态修改在 actions 中完成，或直接在组件中修改 `store.xxx = value`
3. **storeToRefs 必要性**：解构 store 的 state/getters 时必须使用 `storeToRefs()` 保持响应性
4. **actions 可直接解构**：actions 是普通函数，可以直接从 store 解构
5. **跨模块访问**：Pinia 中通过导入其他 store 的 `useXxxStore()` 实现跨模块访问
6. **命名空间简化**：Vuex 的 `namespaced: true` 在 Pinia 中通过 store ID 天然支持
7. **TypeScript 友好**：Pinia 天然支持 TypeScript，`state()` 返回类型即可推导
8. **$reset 方法**：Pinia 内置 `$reset()` 方法快速重置 state（对应 Vuex 的 `replaceState`）
9. **$patch 批量更新**：支持 `store.$patch({ key: value })` 批量更新
10. **$subscribe 监听**：使用 `store.$subscribe()` 替代 Vuex 的 `store.subscribe()`
11. **插件迁移**：如有 Vuex 插件（日志、持久化等），迁移到 Pinia 的同功能插件
12. **避免 this 陷阱**：Options API 中在 `setup` 外部使用 `useXxxStore()` 时需要注意 this 上下文
